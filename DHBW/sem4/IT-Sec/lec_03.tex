\subsubsection{Anfordderungen an kryptographischen Verfahren}

Der Angreifer darf alles über das System wissen. Die Sicherheit darf nur von dem Schlüssel selbst abhängig sein.

Die Gute soll groß sein:

\begin{enumerate}
    \item Schlüsselraum soll sehr groß sein.
    \item Brute Force solle nicht mit Praktikablen aufwand möglich sein.
    \item Ein Analytisches Verfahren soll nicht möglich sein. Somit nur Brute Force möglich.
\end{enumerate}

\subsubsection{Basis: Faktorisierung}

Die vermutete "Einweg-Funktion" ist mit Faktorisierung:

\[
f(p, q) = P * q = n
\]

Und die umkehrabbildung ist schwer:

\[
f^{-1}(n) = p * q
\]

\subsubsection{Basis: Logarithmus}

Oder auch per Logarithmus:

\[
f(x) = g^x \text{ mod } p = y
\]

Mit der schweren Entschlüsselung

\[
f^{-1}(y) = \log_g(y) \text{ mod } p
\]

\subsection{Stromchiffren 2}

In einem WLAN wird WEP eingesetzt, um das Netz "abzusichern". Zur Verschlüsselung verwendet WEP die Stromchiffre RC4, die als Eingabe einen Initialisierungsvektor IV und einen symmetrischen Schlüssel K erhält, um einen Schlüsselstrom RC4(IV | K) zu generieren. Da der IV nur eine Länge von 24 Bit hat, kommt es nach einiger Zeit zu Kollisionen.

a) Welche Folgen hat dies für die Verschlüsselung, wenn ein IV doppelt verwendet wird? Begründen Sie ihre Antwort.

b) Skizziere einen Angriff unter folgender Annahmen:

\begin{enumerate}
    \item $C_1=M_1 RC4(IV \mid K)$
    \item $C_2=M_2 RC4(IV \mid K)$
    \item $M_1$ sei bekannt.
\end{enumerate}

\textbf{Lösung:}

a) Bei gleichem $IV$ und gleichem Schlüssel $K$ entsteht derselbe Schlüsselstrom
$S = RC4(IV \mid K)$. Damit gilt:

$$C_1 = M_1 \oplus S \quad\text{und}\quad C_2 = M_2 \oplus S$$

XORt man beide Chiffretexte, fällt der Schlüsselstrom weg:

$$C_1 \oplus C_2 = M_1 \oplus M_2$$

Dadurch geht Vertraulichkeit verloren: Zusammenhänge der Klartexte werden sichtbar,
und mit Zusatzwissen (z.B. bekannte Textteile) lassen sich Nachrichten rekonstruieren.

b) Angriff bei bekanntem $M_1$ (Known-Plaintext-Angriff):

$$S = C_1 \oplus M_1$$

Damit kann der zweite Klartext direkt berechnet werden:

$$M_2 = C_2 \oplus S = C_2 \oplus C_1 \oplus M_1$$

Also reicht ein bekanntes Klartext-Chiffretext-Paar bei wiederverwendetem $IV$ aus,
um weitere Pakete mit demselben $IV$ zu entschlüsseln.

\subsection{Message-Authentication-Codes}

Hash-Funktionen werden oft in MAC-Verfahren eingesetzt, um die Integrität und Authentizität von Daten zu gewährleisten. Bei einem solchen Keyed-Hash werden die Daten mit einem symmetrischen Schlüssel $k_{AB}$ konkateniert und dann mit der Hash-Funktion $H$ gehasht. Die Nachricht $x$ wird im Rahmen der Hashfunktion zu Blöcken gleicher Größe unterteilt, d.h. es gilt $x = (x_1, x_2, ..., x_n)$.

Man unterscheidet zwei Varianten:

Secret-Prefix: $h = MAC(m, k_{AB}) = H(k_{AB} | m)$

Secret-Suffix: $h = MAC(m, k_{AB}) = H( m | k_{AB})$

\begin{itemize}
    \item \textbf{Secret-Prefix} $H(k\mid m)$: Bei Merkle-Damgard-Hashfunktionen ist ein Length-Extension-Angriff möglich. Aus $(m, H(k\mid m))$ kann ein Angreifer ein gültiges Tag für $m\parallel m'$ konstruieren, ohne $k$ zu kennen.
    \item \textbf{Secret-Suffix} $H(m\mid k)$: Kein klassischer Length-Extension-Angriff, aber das Verfahren ist nicht als sicherer MAC bewiesen und kann bei schwacher Kollisionsresistenz angegriffen werden (z.B. über Kollisionen in $m$ mit gleichem Tag).
    \item \textbf{Warum HMAC?} HMAC nutzt zwei Schlüsselableitungen mit innerem/äußerem Padding:
    $$\mathrm{HMAC}_k(m)=H\big((k\oplus opad)\parallel H((k\oplus ipad)\parallel m)\big).$$
    Dadurch wird die einfache Verkettung vermieden; insbesondere sind bekannte Extension-Angriffe auf die naive Konstruktion nicht direkt anwendbar.
\end{itemize}

\subsection{Public-Key-Verfahren - RSA}

Wir haben den public key (e=5, n=21) und den private key (d=17)

a) Wir wollen die Zahl 5 verschlüsseln:

\[
5^{e} \text{ mod } n = 5^5 \text{ mod } 21 = 3.125 \text{ mod } 21 = 17
\]

b) Ich erhalte die Nachricht 11:

\[
11^{d} \text{ mod } n = 11^{17} \text{ mod } 21 = 2
\]

c) Wir erhalten zwei Nachrichten mit Signatur. Wie sieht die Authentizität aus?

\begin{enumerate}
    \item Die Zahl 3 mit Signatur 12
    
    Die Signatur kann mit $3^{d} \text{ mod } n = 3^{17} \text{ mod } 21 = 12$ geprüft werden. Da 12 der Signatur entspricht ist das der korrekte Absender.
    \item Die Zahl 4 mit Signatur 15
    
    $4^{d} \text{ mod } n = 4^{17} \text{ mod } 21 = 16 \neq 15$. Somit kommt diese Nachricht von einem Falschem Absender 
\end{enumerate}

d) Wie sehen p und q aus mit $n = p * q$?

Die Primfaktorzerlegung von $n = 21 = 3 * 7$ somit den Mathematischen Angriff durchführen.

Mit x und y einer Beispielnachricht 5 und 17

\begin{align}
    \phi(n) = (p-1)(q-1) = 2*6 = 12\\
    d^{-1} = e \text{ mod } \phi(n) = 5 \text{ mod } 12 = 5\\
    x = y^d \text{ mod } n = 17^17 \text{ mod } 21 = 5
\end{align}

Somit korrekt den private key errechnet (d = 17), wobei ein (d = 5) hier auch funktioniert.
