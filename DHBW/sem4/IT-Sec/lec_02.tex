\section{Sicherheitsbedrohungen}

\know{Klausurfrage}{Welche Schutzziele gibt es?
\begin{enumerate}
    \item Vertraulichkeit Confidentiality
    \item Integrität Integrity
    \item Verfügbarkeit Availability
    \item Verbindlichkeit Accountability
    \item Authentizität Authenticity
    \item Privatheit Anonymität Nicht-Verfolgbarkeit Privacy
\end{enumerate}}

Eine Bedrohung des Systems zielt darauf ab, eine oder mehrere Schwachstellen oder Verwundbarkeiten auszunutzen, um ein Schutzziel zu gefährden.

Es gibt mehrere Klassifikation von Bedrohungen:

\begin{enumerate}
    \item Höhere Gewalt
    \item Fahrlässigkeit
    \item Vorsatz
    \item technisches Versagen
    \item Organisatorische Mängel
\end{enumerate}

\subsection{Angriff}

Unter einem Angriff verstehen wir einen nicht autorisierten Zugriff auf ein Asset.

\subsubsection{Passiver Angriff}

Sniffing-Angriff - ausspähen von z.B. Passwörter und Zugangsdaten - Schutzziel Vertraulichkeit verletzt.

Verhindern nur durch Verwendung von Kryprographie.

\subsubsection{Aktiver Angriff}

Verändern, Erzeugen oder Entfernen von Datenpaketen, wie auch unautorisiertes Schreiben in Dateien.

\begin{enumerate}
    \item Buffer-Overflows-Exploit
    \item Universelle Interpretierbarkeit von Daten, Code
    \item Social Engineering
\end{enumerate}

\subsubsection{Buffer-Overflows-Exploit}

Funktioniert so, indem wir den Buffer mit Informationen überschreiben, sodass wir zugriff auf ein System bekommen.

Wenn es gelingt einen Rücksprungbefehl/ Adresse auf einen Sinnvollen Bereich zeigen zu lassen, oder sogar auf einen Bereich, wo ich selbst schon Quelltext reingeschrieben habe. So könnt ein Angreifer beliebigen Code ausführen.

Ein Beispiel was ich kenne ist Arbitrary Code Execution von \href{https://tasvideos.org/8991S}{Super Mario Bros}

\subsection{Malware}

Malware = Malicious Software

\subsubsection{Viren}

Selbstproduzierend und benötigt einen Host, wo dieser ausgeführt werden kann.

Zum Beispiel basic scripte in z.B. Word oder Excel.

\subsubsection{Würmer}

Ist eine spezielle Form von Viren. Sie sind eigenständig und aktiv. Sie brauchen keinen Host.

Können z.B. von sich aus Emails an eine Adressliste senden.

Muss ausgeführt werden.

\subsubsection{Trojanische Pferde}

Kommen durch Programme, die im Hintergrund noch was anderes Machen.

Loggen z.B. Passwärter, Tokens, usw.

Funktionen:

\begin{enumerate}
    \item Exploit - Ziel Rechte zu bekommen
    \item Backdoor - Über einen anderen Nutzer die Kontrolle zu bekommen
    \item Keylogger - Logging von Keyboard Nutzung
    \item Ransom - Blockieren der Nutzung gegen Lösegeld
    \item Fake Antivirus - Geld erpressen durch Panikmache
\end{enumerate}

\subsection{Angriffe auf Netzwerkkommunikation}

\subsubsection{Denial of Service (DoS)}

Unterbrechnung der Netzwerkverbindung - Schutzmittel Verfügbarkeit verletzt.

Häufig durch eine Überflutung von Anfragen.

Es kann auch von mehreren Orten gleichzeitig passieren. Dann ist es \textbf{Distributed Denial of Service (DDoS)}

\subsubsection{Sniffing}

Einfaches Abhören - Verletzt Vertraulichkeit

\subsubsection{Man in the Middle (MiM)}

Abfangen und Modifizieren - Schutzziel Integrität verletzt

\subsubsection{Spoofing}

Angreifer X schickt Daten an Knoten B täuscht vor, dass die Daten von Knoten A kommen würden.

Das Spoofing gibt es auf allen Schichten des OSI-Modells.

Varianten:

\begin{enumerate}
    \item ARP-Spoofing - Es wird versucht das Mapping von MAC auf IP zu manipulieren (über Broadcast)
    \item IP-Adress-Spoofing - IP-Packet mit gefälschter IP Adresse versenden
    \item Session-Hijacking
    \item DNS Spoofing - (Es gab einen großen Teil zu DNS)
    \item E-Mail-Spoofing
    \item Web-Spoofing
\end{enumerate}

\subsection{Cross-Site-Scripting}

\href{https://cwe.mitre.org/data/definitions/79.html}{Cross-Site-Scripting}

Varianten:

\begin{enumerate}
    \item Reflected XSS / Non-Persistent XSS
    Empfängt manipulierte Daten und gibt eine Antwort
    \item Stored XSS / Persistent XSS
    Speichert manipulierte Daten und gibt eine Antwort auf anfrage
    \item DOM-based XSS
\end{enumerate}

\subsubsection{Cross-Site Request Forgery (CSRF)}

Dieser nutzt aus, das ein Nutzer in mehreren Tabs angemeldet ist.

Vor diesen Angriff kann  geschützt werden, indem jede Anfrage mit einem token versendet wird. Nur bei korrektem token wird die Anfrage genehmigt.

\subsection{Common Weakness Enumeration (CWE)}

\href{https://cwe.mitre.org}{CWE} ist eine Aufzählung von Schwachstellentypen. Sind nach ID geordnet.

Die hat auch ein Scoring System \href{https://cwe.mitre.org/cwss}{CWSS} oder auch \href{https://nvd.nist.gov/cvss.cfm}{CWSS Calculator}

Themen:

\begin{enumerate}
    \item Common Weakness Enumerator (CWE)
    \item Common Attack Pattern Enumeration and Classification (CAPEC)
    \item COmmon Weakness Scoring System (CWSS)
    \item Common Vulnerabilities and Exposures (CVE)
    \item Common Vulnerabilities Scoring System (CVSS)
    \item Common Platform Enumeration (CPE)
    \item Common Configuration Enumeration (CCE)
\end{enumerate}

\subsection{Heartbleed}

Heartbleed war eine Extension, welche die Verbindung zwischen Server und Client prüft. Es empfängt eine Nachricht mit Angabe der Länge und gibt alles aus dem Speicher zurück, bis zur Länge.

Der Heartbleed Bug umfasst mehrere Sicherheitslücken gelistet unter CWE. Einer ist der \href{https://cwe.mitre.org/data/definitions/125.html}{Out-of-bounds Read} Prinzipiell werden hierbei Daten aus dem Buffer ausgelesen, welche eigentlich nicht nach außen gezeigt werden sollen, wie z.B. Tokens.

Schutzziele die verletzt wurden:

\begin{enumerate}
    \item Vertraulichkeit
    \item (Integrität)
    \item Privatheit, wenn der private key ausreicht um die Identität zu klauen
    \item Authentizität
\end{enumerate}

Er wurde als Katastrophal bezeichnet, da hier tokens aller Art und private Informationen aus dem Buffer gelesen werden konnten. Angreifer bleiben \textbf{unbekannt} und hinterließen \textbf{keine Spuren}. Die Extension war \textbf{überall} in Anwendung.

\subsection{Apple Goto Fail}

\href{https://www.heise.de/hintergrund/Diverse-Personen-arbeiten-daran-diesen-Angriff-in-ihre-Tools-einzubauen-2121951.html?seite=all}{Artikel} zum nachlesen.

Bezieht sich auf den Handshake von SSL-Verbindungen (Diffie-Hellman-Schlüsselaustausch) Um einen Man in the Middle Angriff zu verhindern, wird die Nachricht vom Server signiert. Durch das zusätzliche "goto fail;" wird die Signatur aber nie überprüft. 

Schutzziele die verletzt wurden:

\begin{enumerate}
    \item Vertraulichkeit
    \item Integrität
    \item Privatheit
    \item Authentizität
\end{enumerate}

\subsection{Vertrauen in CA}

\href{https://www.heise.de/news/Zertifikats-Klau-Fatale-Sehschwaeche-bei-Comodo-3354229.html}{Artikel} zum nachlesen.

Der verwendete Texterkennungsalgorithmus hatte eine "seeschwäche".

So hatte er Schwierigkeiten, die Ziffer 1 von einem klein geschriebenen L ("l") zu unterscheiden. Das Gleiche galt für für Ziffer 0 und das kleine O ("o"). Bei der Entscheidungsfindung hat der OCR-Algorithmus offenbar die nachfolgenden Zeichen ausgewertet: Folgte auf ein kleines L eine Ziffer, hielt er den Buchstaben für die Zahl 1. Folgte auf eine Null ein Buchstabe, wurde daraus das kleine O. Dadurch wurde etwa aus "@level3.com" die Variante "@leve13.com" und aus "@c0re.sk" wurde "@core.sk".

\href{https://www.heise.de/news/HTML-Injection-Luecke-erlaubte-Zertifikatsklau-bei-Comodo-3282183.html}{Artikel} zum nachlesen.

Hier wurde HTML-Injection in die Bestätigungs-Email umgesetzt. So konnte ein Angreifer einfach für eine Zeilwebseite ein Zertifikat beantragen, welches an den Admin gesendet wurde. Durch die HTML-Injection wurde der Inhalt aber geändert und ein Button "CLick hier to cancel the request" erweitert, welcher beim drücken den Inhalt mit Zertifikat and den Angreifer sendet.
