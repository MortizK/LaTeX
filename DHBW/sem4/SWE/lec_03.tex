\subsection{Nach Fowler}

Die Refactoring Struktur nach Fowler:

\begin{enumerate}
    \item Name
    \item Summary
    \item Motivation
    \item Mechanics
    \item Example
\end{enumerate}

Es gibt mehrere Gruppen:

\begin{enumerate}
    \item Composing Methods
    \item Moving Features Between Objects
    \item Organizing Data: Arrays mit Objekten austauschen, außer wir arbeiten Hardwarenahe.
    \item Simplifying Conditionals: Complex Conditionals in Funktionen umwandeln.
    \item Making Method Calls Simpler
    \item Generalization
\end{enumerate}

\subsubsection{Composing Methods}

Mögliche Refactoringaufgaben unter dieser Gruppe.

\begin{enumerate}
    \item Aus einen Stück Code eine Funktion erstellten.
    \item Aus einer Funktion eine Klasse mit Object erstellen.
    \item TemporäreVariabel ordentliche Namen geben und eine eigene Pro Zweck verwenden.
\end{enumerate}

\subsubsection{Moving Features Between Objects}

\begin{enumerate}
    \item Move Method: Methode wird meist von einer anderen Klasse aufgerufen - Also Verschiebung dorthin.
    \item Move Field: Ein Attribute wird meist von einer anderen Klasse benutzt - Also Verschiebung dorthin.
    \item Ein Klasse enthält mehrere Funktionsbereiche - Aufteilen in kleinere Klassen.
\end{enumerate}

\subsubsection{Making Method Calls Simpler}

\begin{enumerate}
    \item Parameter Object - Statt vielen Parameter, eine Klasse erstellen, welche die Parameter vorgibt.
    \item Hide Method - Sichtbarkeiten von Methoden benutzen
    \item Signaturen anpassen, um Code Clones zu verhindern.
\end{enumerate}

\subsubsection{Generalization}

Hier geht es darum die Vererbungsstruktur zu verbessern. Also Superklassen und Zwischenklassen erstellen/ anpassen. Wobei es auch zu tiefe Vererbungshierarchien gibt.

\begin{enumerate}
    \item Pull Up Field: Attribute an die Superclass abgeben, wenn das Attribut in allen Unterklassen passend ist. Oder halt eine Zwischenklasse erstellen.
    \item Pull Up Method: Methode an die Superclass abgeben.
    \item Extract Superclass
\end{enumerate}

\subsection{HomeController}

In the HomeController I have two functions: \lstinline|private void loadXMLTabContent()| and \lstinline|private void loadPDFTabContent()| which sound similar and are similar. Here I can use \href{https://refactoring.com/catalog/parameterizeFunction.html}{Parameterize Function} To change two functions into one function with appropriate parameters.

In the Tasks Model I have two functions, which do the same formatting of a string. Here I can use \href{https://refactoring.com/catalog/extractFunction.html}{Extract Function} to create \lstinline|toTitleCase| 

\begin{lstlisting}[style=javastyle]
public String getFormatedDifficulty() {
    String diff = difficulty.toString();
    return toTitleCase(diff);
}
public String getFormatedScope() {
    String sc = scope.toString();
    return toTitleCase(sc);
}
private String toTitleCase(String s) {
    return s.charAt(0) + s.substring(1).toLowerCase();
}
\end{lstlisting}

My DataService currently mixes Updating UI and State Handling. This could be refactored with \href{Uhttps://refactoring.com/catalog/extractClass.htmlRL}{Extract Class}. This will may result in two possible Classes: ExamNavigationService and ViewStateService.
