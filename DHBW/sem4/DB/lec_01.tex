\subsection{Begriffe}

Datenbankmodell: Logische Struktur eines DBMS, welches beschreibt in welcher Form Daten strukturiert, modelliert und abgespeichert werden. Beispiel: Hierarchisch, Relational, NoSQL (Key, Value)

Datenbankschema: beschreibt die Struktur des Datenbestands, also die Metadaten. Wie beispielsweise hängen die Tabelle zusammen, welche Integritäts-bedingungen bestehen usw. Beispiel: Die Definition der Attribute der Tabellen. Ein Mitarbeiter hat Name, Vorname, usw.

Datenbanksprache: 

\begin{enumerate}
    \item DDL = Data Definition Language - Anlegen von Tabellen
    \item DML = Data Manipulation Language - Einfügen, Löschen
    \item DQL = Data Query Language - Abfragen
\end{enumerate}

\subsection{Die 3-Ebenen Architektur}

\begin{enumerate}
    \item externe Ebene - Benutzeroberfläche
    \item konzeptionelle Ebene - Tabellen, Beziehungen
    \item interne Ebene - Dateiorganisation und Zugriffspfades
\end{enumerate}

Vorteile dieser Struktur ist die geringere Komplexität der einzelnen Komponenten. 

Wenn eine Änderung vorgenommen wird, muss nur so tief gegangen werden wie das tiefste Element. Zum Beispiel ein neues Tabellenblatt. So muss nur die Benutzeroberfläche und die konzeptionelle Ebene angepasst werden und NICHT die interne Ebene.

\subsection{Normalformen}

Die 1NF darf keine Mengen Attribute haben und Mehrwertige Attribute ($T_1-T_n$)

Die 2NF: Alle Nichtschlüsselattribute dürfen nur durch den gesamten Primär-schlüssel festgelegt werden.

Die 3NF: Wenn es keine transitiven Abhängigkeiten zwischen Nichtschlüssel-attribute und einem Schlüsselkanditaten besteht.

Wir haben eine Formale Prüfung nach 3NF:

\begin{enumerate}
    \item Zerlegen auf die einzelnen FAs
    \item Prüfen ob gilt: entweder Superschlüssel oder prim.
\end{enumerate}

Und es gibt noch die Boyce-Codd Normalform. (Diese gibt es nur in der Theorie)
