\subsubsection{Constraints}

Mögliche Constraints:

\begin{enumerate}
    \item NOT NULL
    \item UNIQUE
    \item PRIMARY KEY
    \item FOREIGN KEY
    \item CHECK \textit{Bedingung}
    \item DEFAULT \textit{wert}
\end{enumerate}

Mitarbeiter mit Constraints:

\begin{lstlisting}
CREATE TABLE Mitarbeiter
(
    Pers_Nr         integer PRIMARY KEY,
    Vorname         varchar,
    Nachname        varchar,
    Geb_Name        varchar,
    Geb_Datum       date,
    Geschlecht      char(1),
    Eintrittsdatum  date,
        PRIMARY KEY (Pers_Nr),
		CHECK (Geschlecht IN ('m', 'w', 'd'))
);
\end{lstlisting}

Für CHECK gibt es beliebig viele Abfragen und Methoden.

Bei einem PRIMARY KEY wird automatisch noch ein Index erstellt. Dies beschleunigt die Suche nach einem PRIMARY KEY.

Manuell kann auch für jedes Andere Attribute ein Index erstellt.

\begin{lstlisting}
CREATE TABLE Mitarbeiter
(
    Pers_Nr         integer,
    Vorname         varchar,
    Nachname        varchar,
    Geb_Name        varchar,
    Geb_Datum       date,
    Geschlecht      char(1),
    Eintrittsdatum  date,
	Skill			varchar(5) default 'PR',
	Gehalt			int,
        PRIMARY KEY (Pers_Nr),
		CHECK (Geschlecht IN ('m', 'w', 'd')),
		CHECK (Gehalt > 20000 and Gehalt < 150000)
);
\end{lstlisting}

\subsubsection{Fremdschlüssel}

Mit einem Fremdschlüssel können wir auf eine andere Vorhandene Tabelle Zeigen. Es benutzt automatisch den Primär Schlüssel der Fremdtabelle.

\begin{lstlisting}
DROP TABLE IF EXISTS  Mitarbeiter;
CREATE TABLE IF EXISTS  Mitarbeiter
(
    Pers_Nr         integer,
    Vorname         varchar,
    Nachname        varchar,
    Geb_Name        varchar,
    Geb_Datum       date,
    Geschlecht      char(1),
    Eintrittsdatum  date,
	Skill			varchar(5) default 'PR',
	Gehalt			int,
	arbeitet_in		varchar(8),
	chef_von		integer,
PRIMARY KEY (Pers_Nr),
CHECK (Geschlecht IN ('m', 'w', 'd')),
CHECK (Gehalt > 20000 and Gehalt < 150000),
FOREIGN KEY(arbeitet_in) REFERENCES Abteilung,
FOREIGN KEY(chef_von) REFERENCES Mitarbeiter
);
\end{lstlisting}

Um zu verhindern, dass wir endlos immer 'tiefer' eine reference durchgeführt werden. Deshalb:

\know{FOREIGN KEY}{Erst im Nachhinein die Referenzen durchführen und zuerst die Reine Struktur der einzelnen Tabelle.}

Dies kann dann mit dem \lstinline|ALTER TABLE| durchführen

\begin{lstlisting}
ALTER TABLE Mitarbeiter
    ADD arbeitet_in     varchar(8),
    ADD FOREIGN KEY (arbeitet_in) REFERENCES Abteilung;
\end{lstlisting}

\subsubsection{Ternäre Beziehung}

Aus S.22: Dies ist eine beliebte Klausuraufgabe! (Aufgabe unten)

\begin{enumerate}
    \item Bestimmen Sie die Kardinalitäten der Ternären Beziehung,
    \item Stellen Sie die vier Relation auf und legen Sie die Schlüssel für die Relation "arbeitet" fest.
    \item Erstellen Sie die Tabelle "arbeitet" in SQL mit allen Constrains.
    \item Prüfen Sie ob die Anforderungen erfüllt sind.
    \item Kann die Trinäre Beziehung in mehrere Binäre Beziehungen umgewandelt werden, ohne das ein Semantikverlust auftritt?
\end{enumerate}

Wir sollen aus den vier Bedingungen den Beziehungsstatus erstellen. Von Ort (1), Projekt (1), Mitarbeiter (n)

\begin{enumerate}
    \item Ein Projekt, Ein Ort hat mehrere Mitarbeiter
    \item Ein Mitarbeiter, Ein Projekt hat Ein Ort
    \item Ein Ort, Ein Mitarbeiter hat ein Projekt
    \item Ein Mitarbeiter, Mehrere Projekte hat mehrere Orte
\end{enumerate}

Nun sollen wir für die "arbeitet" Tabelle bilden.

arbeitet (partielle Funktion):

\begin{enumerate}
    \item MA x Ort $\rightarrow$ Projekt
    \item MA x Projekt $\rightarrow$ Ort
\end{enumerate}

arbeitet1 (Relation): \{[\underline{Mitarbeiter ID, Projekt ID}, Ort ID]\}

arbeitet1 (Relation): \{[\underline{Mitarbeiter ID, Ort ID}, Projekt ID]\}

\begin{lstlisting}
CREATE TABLE IF NOT EXISTS arbeitet
(
	Pers_Nr integer NOT NULL,
	Projekt_Nr integer NOT NULL,
	Ort_Nr integer,
        PRIMARY KEY (Pers_Nr, Projekt_Nr),
        FOREIGN KEY (Pers_Nr) REFERENCES Mitarbeiter,
        FOREIGN KEY (Projekt_Nr) REFERENCES Projekt,
        FOREIGN KEY (Ort_Nr) REFERENCES Ort,
        CONSTRAINT PersOrt UNIQUE (Pers_Nr, Ort_Nr) 
); 
\end{lstlisting}

Wir können eine Trinäre Beziehung auch mit mehreren Binären darstellen. Hierzu muss die Struktur und die Kardinalitäten angepasst werden.

Andere Aufgabe:

\begin{enumerate}
    \item Ein Thema wird höchstens einmal vergeben
    \item Ein Assistent kann höchstens einmal promovieren
    \item Professoren können mehrere Assistenten und unterschiedliche Themen betreuen
\end{enumerate}

Einfach mal so eine Tabelle

\begin{center}
    \begin{tabular}{c|c|c|c}
        Prof & Assistent & Thema & erlaubt\\
        \hline
        1000 & 1 & A & \\
        2000 & 2 & B & \\
        3000 & 1 & C & nein\\
        2000 & 4 & A & nein\\
        1000 & 3 & D & \\
    \end{tabular}
\end{center}

Um das jetzt nun umzusetzen brauchen wir einen Primärschlüssel (entweder Assistent oder Thema) und noch einen UNIQUE CHECK auf das jeweils andere. Somit kann ein Assistent höchstens einmal Promovieren und jedes Thema wird nur einmal vergeben.

Primärschlüssel fällt auf den Key nachdem häufiger gesucht wird, da ein Index erstellt wird.

\subsubsection{DELETE}

Damit DELETE ermöglicht wird bei Beziehungen schon angegeben, was passieren soll wenn DELETE wird:

\begin{lstlisting}
CREATE TABLE Mitarbeiter
(
    ...
    FOREIGN KEY (arbeitet_in) REFERENCES Abteilung
    on Delete set NULL;
)
\end{lstlisting}

Automatisch wird die regel auf \lstinline|RESTRICT| gesetzt. Also das Delete wird verhindert. Unser Dozent plädiert dazu auch ein \lstinline|RESTRICT| active zu setzen um den Überblick und die intention zu behalten.
